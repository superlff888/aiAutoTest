# 需求拆分技能 (Requirement Splitter)

## 1. 技能定位

本技能专注于**阶段1-2：文档解析 + 需求梳理**，负责：

1. **自动解析**：检测输入格式，PDF/Word/PPT 自动调用 MinerU 转 md（`extract` 模式，支持表格/公式识别）
2. 按章节结构识别功能模块
3. 拆分需求单元并统一编号
4. 提取需求关联的图片
5. 生成 JSON 格式的结构化需求文档

**包含**：文档解析（通过 MinerU）+ 需求拆分

**不包含**：测试点提取（由 testpoint-extractor 技能负责）

***

## 2. 输入输出规范

### 2.1 输入

| 参数           | 类型   | 说明            |
| ------------ | ---- | ------------- |
| `input_doc`  | path | 需求文档路径，支持 PDF/Word/PPT/md 格式 |
| `output_dir` | path | 输出目录路径        |

### 2.1.1 支持的输入格式

| 格式 | 扩展名 | 处理方式 |
|------|--------|----------|
| Markdown | `.md` | 直接进入章节解析 |
| PDF | `.pdf` | 调用 `mineru-open-api extract` 转 md + images |
| Word | `.docx`, `.doc` | 调用 `mineru-open-api extract` 转 md + images |
| PowerPoint | `.pptx`, `.ppt` | 调用 `mineru-open-api extract` 转 md + images |
| HTML | `.html` | 调用 `mineru-open-api extract` 转 md |

### 2.1.2 MinerU 文档解析（自动触发）

当输入格式 **不是 md** 时，自动执行以下流程：

1. **检测格式**：检查 `input_doc` 扩展名
2. **调用 MinerU**：
   ```bash
   mineru-open-api extract "{input_doc}" -o workspace/source/output/mineru_doc/{项目名称}_{hash}/ --model pipeline
   ```
   - 使用 `extract` 模式（支持表格/公式识别，无页数限制）
   - 使用 `--model pipeline`（无幻觉风险，适合需求文档的准确性和可靠性）
   - Token 已配置在 `~/.mineru/config.yaml`，无需额外认证
3. **获取输出**：MinerU 输出 md 文件和 images/ 目录
4. **衔接解析**：将生成的 md 路径作为后续章节解析的输入

> **重要**：MinerU 解析完成后，生成的 md 文件中的图片引用路径为相对路径（如 `![](images/xxx.png)`），
> 本技能后续的图片复制逻辑需要适配 MinerU 的输出目录结构。

### 2.2 输出目录结构

```
.claude/output/aiAutoTester/          # output_dir 默认值
├── {项目名称}/
│   ├── 03_requirements/
│   │   ├── 01_第{章}章_{章节名}/
│   │   │   ├── requirements.json   # 需求文档 (JSON)
│   │   │   └── images/             # 该章节关联的图片
│   │   ├── 02_第{章}章_{章节名}/
│   │   │   └── ...
│   │   └── README.md               # 章节索引（人可读）
│   
```

## 3. JSON 需求文档格式

### 3.0 可测试需求的核心要求（重要）

#### 3.0.1 UI 操作类需求

**每个需求必须包含以下要素，否则不能作为独立需求提取：**

| 要素        | 说明          | 必须性 |
| --------- | ----------- | --- |
| **触发条件**  | 什么情况下触发这个功能 | 必须  |
| **输入**    | 需要什么数据/参数   | 必须  |
| **动作/操作** | 用户或系统做什么操作  | 必须  |
| **预期输出**  | 期望的结果是什么    | 必须  |
| **验证点**   | 如何验证功能正确    | 必须  |

#### 3.0.2 计算规则/公式类需求

**公式/规则类需求的要素（同样必须包含，但侧重不同）：**

| 要素        | 说明          | 示例 |
| --------- | ----------- | ---- |
| **触发条件**  | 满足什么条件时应用此公式 | "当 {参数A} {比较符} {参数B} × {系数} 时" |
| **输入参数**  | 公式涉及的输入变量   | {参数1}, {参数2}, {参数3} 等 |
| **计算公式**  | 完整的计算规则/公式   | {输出} = ({变量A} × {系数} - {变量B}) × Max({变量C} - {变量D}, 0) × {系数2} |
| **预期输出**  | 计算结果是什么     | {计算结果变量}（{单位}） |
| **边界条件**  | 特殊规则/阈值      | "{条件}时该时段不回收" |

> **如果一个需求无法回答上述5个问题，则不是可测试的需求，不应提取。**

### 3.1 单个需求单元结构

```json
{
  "requirement_id": "REQ-CHX-001",
  "requirement_name": "新增{业务对象}信息",
  "type": "功能需求 (FR)",
  "module": "{业务对象}管理",
  "chapter": X,
  "source_page": "第 XX 页",

  "description": "{角色}可以在{场景}下新增{业务对象}信息",

  "testable_elements": {
    "trigger": "{触发条件}",
    "inputs": ["{输入参数1}", "{输入参数2}"],
    "action": "{执行动作}",
    "expected_output": "{预期输出}",
    "verification_points": [
      "{验证点1}",
      "{验证点2}"
    ]
  },

  "detailed_description": "【用例 UC-XXX-001】\n\n{完整的需求原文描述}",
  "acceptance_criteria": [
    "{验收标准1}",
    "{验收标准2}"
  ],

  "related_images": [
    {
      "filename": "xxx.jpg",
      "description": "{图片描述}",
      "path": "images/xxx.jpg"
    }
  ],
  "source_paragraphs": ["Para-XXX-001"]
}
```

### 3.2 章节需求 JSON 结构

```json
{
  "chapter": {
    "chapter_number": 3,
    "chapter_title": "XXXXXX",
    "total_requirements": 25
  },
  "modules": [
    {
      "module_name": "XXXXXX",
      "requirement_count": 5,
      "requirements": [
        // 每个 requirement 包含完整 detailed_description
      ]
    },
    {
      "module_name": "XXXXX",
      "requirement_count": 8,
      "requirements": [
        // 每个 requirement 包含完整 detailed_description
      ]
    }
  ]
}
```

### 3.3 全局汇总 JSON 结构

```json
{
  "project_name": "{项目名称}",
  "document_version": "V1.0",
  "total_chapters": 1,
  "total_requirements": 8,
  "total_images": 0,
  "chapters_summary": [
    {
      "chapter_number": "5.6.2.2",
      "chapter_title": "{章节名}",
      "module_count": 4,

      "requirement_ids": ["REQ-5622-001", "REQ-5622-002", ...]
    }
  ],
  "all_requirements": [
    {
      "requirement_id": "REQ-5622-001",
      "chapter_number": "5.6.2.2",
      "chapter_title": "{章节名}",
      "module": "{模块名}",
      "name": "{需求名称}"
    }
  ]
}
```

## 4. 需求拆分规则

### 4.1 章节识别规则

1. 识别文档中的**内容一级标题**（`#`）作为章(如果该章节下面没有具体的需求则跳过)
2. 识别文档中的功能分类 作为功能模块的名称
3. 每个一级标题下的内容属于同一章

### 4.2 需求单元识别规则

#### 4.2.1 需求识别原则（重要）

**必须提取的功能性需求：**

- 有明确动作动词的需求（如：新增、删除、修改、查询、展示、使用、发放、导出等）
- 可以设计具体测试用例的需求
- 有明确输入、输出、触发条件的需求
- 【用例UC-xxx】描述的功能点
- 【需求场景】中描述的具体功能

**不应提取的非功能性内容（跳过）：**

- 背景描述、业务背景、项目背景
- 概念说明、术语定义、业务前提与假设
- 章节/小节概述性文字
- 业务流程概述（非具体功能点）
- 通用说明、注意事项（无具体功能操作）
- 列表形式的属性/字段说明（无增删改查动作）
- "系统应支持..."开头但无具体操作描述的概括性内容

**判断标准：**

> 如果一个需求**不能设计出具体的测试用例**（没有明确的操作动作、没有可验证的输入输出），则不应作为独立需求提取。

#### 4.2.2 识别关键词信号

| 应提取（有关键动作） | 不应提取（仅有描述性文字） |
| ---------- | ------------- |
| 新增、创建、增加   | 概述、说明、指、表示    |
| 删除、移除、注销   | 背景、概念、定义、前提   |
| 修改、更新、变更   | 适用于、一般、用于、属于  |
| 查询、检索、搜索   | 包括、含有、具备、拥有   |
| 展示、显示、呈现   | 满足、实现、提供、支持   |
| 使用、应用、调用   | 通过、根据、按照、随着   |

#### 4.2.3 字段说明

| 字段                    | 说明                   | 来源                   | 完整性要求                 |
| --------------------- | -------------------- | -------------------- | --------------------- |
| requirement\_id       | 唯一标识，如 `REQ-5622-001` 或 `REQ-CH3-001` | 自动生成                 | -                     |
| requirement\_name     | 简洁的功能描述              | 从文本中提取               | -                     |
| type                  | 功能需求/性能需求/安全需求等      | 根据内容判断               | -                     |
| module                | 所属的功能模块              | 根据内容判断               | -                     |
| source\_page          | 在原文档中的页码位置           | 从文本中解析               | -                     |
| description           | 需求简述（一句话概括）          | 从文本中提取               | -                     |
| detailed\_description | **完整原始需求描述**         | 保留原文所有内容             | **必须完整保留原文所有细节，不得截断** |
| acceptance\_criteria  | 验收标准列表               | 从文本中提取               | -                     |
| related\_images       | 关联的图片列表              | 解析 `![](images/xxx)` | -                     |
| source\_paragraphs    | 源段落锚点列表              | 根据行号估算               | -                     |

> **重要**：`detailed_description` 字段必须完整保留从原文档中提取的需求内容，包括：
>
> - 所有功能点描述
> - 所有业务规则
> - 所有条件限制
> - 所有数据字段要求
> - 原文中的所有细节
>
> 不得省略、摘要或简化原文内容。

### 4.3 需求拆分稳定性规则（重要）

为保证多次运行结果一致性，必须遵守以下规则：

#### 4.3.1 顺序保证

- **严格按照原文段落顺序提取需求**，不得打乱顺序
- 需求ID编号必须与原文段落出现顺序一致
- 不允许跨段落合并多个需求为一个

#### 4.3.2 拆分粒度标准

- **一个【用例UC-xxx】= 一个需求单元**
- **一个【需求场景】= 一个需求单元**
- **一个独立的功能点标题 = 一个需求单元**
- 不允许将多个独立需求合并为一个
- 不允许将一个完整需求拆分多个

#### 4.3.3 模块归属映射规则

**通用模块识别规则**（按优先级执行）：

| 优先级 | 识别规则                | 模块归属         |
| --- | ------------------- | ------------ |
| 1   | 包含"新增/创建/增加/增"关键词   | 后台管理         |
| 2   | 包含"删除/移除/注销/删"关键词   | 后台管理         |
| 3   | 包含"修改/更新/变更/改"关键词   | 后台管理         |
| 4   | 包含"查询/检索/搜索/查"关键词   | 后台管理         |
| 5   | 包含"展示/显示/呈现"关键词     | 前台/用户界面      |
| 6   | 包含"使用/应用/调用"关键词     | 前台/用户界面      |
| 7   | 包含"查看/浏览"关键词        | 前台/用户界面      |
| 8   | 包含"导出/报表/统计"关键词     | 报表功能         |
| 9   | 包含"新增.\*类型/创建.\*类型" | {业务}\_类型管理   |
| 10  | 以上都不匹配              | 功能模块（从功能名推断） |

**模块名称生成规则**：

```
当识别到业务关键词时，模块名 = {业务关键词}+"管理"
示例：
- 新增电价类型 → 模块：电价类型管理
- 新增客户信息 → 模块：客户信息管理
- 新增结算规则 → 模块：结算规则管理
```

> **强制规则**：模块归属按上述优先级规则自动判断，不得硬编码特定业务术语

#### 4.3.4 需求名称生成规则

```
格式：{动作}+{业务对象}
示例：
- 计算{结果变量} → "计算{结果变量}"
- 查询结算电量 → "查询结算电量"
- 验证{业务变量} → "验证{业务变量}"
- 返还{结果变量} → "返还{结果变量}"
```

#### 4.3.5 描述提取规则

**description格式（强制）**：

```
格式：[角色]可以在[场景]下[执行动作]来[实现目标]
示例：{角色}可以在{场景}下{执行动作}来{实现目标}
```

**description不满足测试需求时的处理**：

- 如果原文描述不包含完整的主语+动作+宾语，补全但不改变原意
- 如果描述过于笼统（如"系统应支持..."），需细化具体操作
- 如果描述是业务规则而非功能操作，不提取为独立需求

**detailed\_description**：保留原文所有内容，不删减。必须包含：

- 完整的计算公式（含全部参数和变量）
- 具体的业务规则
- 所有的条件限制（含阈值、系数）
- 参数定义和取值范围

#### 4.3.6 testable\_elements强制要求

每个需求**必须包含**testable\_elements字段，否则该需求不可用。根据需求类型选择模板：

**模板A：UI 操作类**
```json
"testable_elements": {
  "trigger": "触发条件：什么操作或事件触发了这个功能",
  "inputs": ["输入1", "输入2"],
  "action": "执行动作：用户或系统做了什么",
  "expected_output": "预期输出：期望的结果是什么",
  "verification_points": ["验证点1", "验证点2"]
}
```

**模板B：计算规则/公式类**
```json
"testable_elements": {
  "trigger": "触发条件：满足什么条件时应用此公式，如'当 {参数A} {比较符} {参数B} × {系数} 时',
  "inputs": ["{参数1}", "{参数2}", "{参数3}", "{系数1}", "{系数2}"],
  "action": "计算公式：{输出} = ({变量A} × {系数} - {变量B}) × Max({变量C} - {变量D}, 0) × {系数2}",
  "expected_output": "输出结果：{计算结果变量}",
  "verification_points": ["{边界条件}时输出为0", "{系数1}取值符合规定", "{系数2}=X"]
}
```

**如果无法提取testable\_elements，该需求应标记为"非功能需求"或跳过**。

#### 4.3.6 一致性校验

完成拆分后必须校验：

1. 需求ID连续不间断（如：REQ-5622-001, REQ-5622-002... 或 REQ-CH3-001, REQ-CH3-002...）
2. 每个模块的requirement\_count与实际requirements数组长度一致
3. 章节total\_requirements等于所有模块requirement\_count之和

### 4.4 需求编号规则

```
REQ-{章节编号}-{序号:3位}
例：REQ-5622-001  表示第5.6.2.2节的第1个需求
例：REQ-CH3-001   表示第X章的第1个需求
```

## 5. 图片关联规则

### 5.1 图片识别

1. 解析 md 文档中的图片引用语法：`![](images/xxx.jpg)` 或 `![Image](url)`
2. 记录图片的原始路径和文件名
3. 判断图片属于哪个语义分组（根据 Markdown 缩进层级和文档结构判断）

### 5.2 图片归属判定（同层级分组规则）

**核心原则：图片向上查找，关联到与它同级的语义分组节点，同一分组下的所有文本 + 所有图片作为一个整体。**

#### 5.2.1 判定步骤

```
遇到一张图片
    │
    ▼
Step 1: 确定图片的 Markdown 缩进层级
    │
    ▼
Step 2: 向上查找同一缩进层级的父节点
    │   （即：最近的、缩进层级 ≤ 图片层级的标题或列表项）
    │
    ▼
Step 3: 确定该父节点下的所有内容块
    │   包括：所有子项文本 + 所有图片
    │
    ▼
Step 4: 将图片关联到该父节点（而非仅前一个段落）
```

#### 5.2.2 两种典型场景

**场景A：多个子项共享一张图（整组关联）**

图片出现在一组子项描述之后，描述的是整个分组的完整界面：

```markdown
- 平铺分析-按时段平铺：            ← 父节点（分析维度）
    - 短期功率预测图（96点颗粒度）  ← 子项文本1
    - 偏差分析图（96点颗粒度）      ← 子项文本2
    - 气象分析图                    ← 子项文本3

![截图]                            ← 图片：属于"按时段平铺"整个维度
```

→ 图片关联到"平铺分析-按时段平铺"父节点
→ `related_images` 包含该图片，且该需求单元的 `detailed_description` 包含全部3个子项文本

**场景B：每个编号项各有自己的图（逐项关联）**

图片紧跟在每个编号项之后，只描述该编号项的内容：

```markdown
1. 租户入驻新增发电侧...           ← 编号项1
![截图1]                           ← 图片：仅属于编号项1

2. 系统角色权限管理...             ← 编号项2
![截图2]                           ← 图片：仅属于编号项2
```

→ 截图1关联到编号项1，截图2关联到编号项2
→ 各自生成独立的需求单元

#### 5.2.3 判断标准

| 判定条件 | 归属结果 |
|----------|----------|
| 图片与前面的子项列表**同缩进层级**，且子项列表是某个父节点的**完整描述** | 图片关联到**父节点**（场景A） |
| 图片紧跟在某个**独立编号项**之后，且该编号项自身是完整的 | 图片关联到**该编号项**（场景B） |
| 同一编号项后出现**多张图片** | 多张图片都关联到该编号项 |
| 图片前面**没有文本**（连续多张图片） | 所有连续图片关联到同一父节点 |

## 6. 验收标准提取规则

### 6.1 识别关键词

验收标准通常出现在以下关键词后：

- "验收标准"
- "完成条件"
- "验收条件"
- "验证条件"
- "上线标准"
- "功能要求"
- "需求"

### 6.2 验收标准格式要求

**好的验收标准示例**：

```
✅ 管理员新增用户时，用户名不能为空
✅ 输入有效的查询条件，系统返回对应的信息列表
✅ 删除已有关联数据的记录时，系统提示无法删除
✅ 当输入数据格式错误时，系统提示相应的错误信息
```

**不合格的验收标准示例**：

```
❌ 系统应支持结算管理功能（太笼统）
❌ 保证计算精度（无法验证）
❌ 符合相关法规要求（无法验证）
```

### 6.3 验收标准提取规则

1. 查找包含验收关键词的段落
2. 提取**具体可测量的条款**（不是概述性描述）
3. 每个验收标准必须包含：
   - 具体的条件（如：输入、状态、前提）
   - 可验证的结果（如：显示、返回、提示）
4. 如果原文没有明确的验收标准，从detailed\_description中**反向推导**可测试的验收标准

### 6.4 验收标准JSON格式

```json
"acceptance_criteria": [
  "管理员可以新增用户信息，保存后列表显示新增的用户",
  "用户名为空时，系统提示'用户名不能为空'",
  "已有关联数据的用户不可删除",
  "用户状态变更后列表中状态同步更新"
]
```

### 6.5 从业务规则推导验收标准

如果原文只描述了业务规则（如计算规则），需推导出可测试的验收点：

```
原文：{条件阈值}时，按{计算规则}进行偏差回收

推导验收标准：
- {条件}，系统计算{结果变量}
- {条件}，系统不进行偏差回收
- {边界条件}时，系统不回收
```

## 9. 执行流程

```
0. 文档格式检测与解析（新增 - 自动触发）
   ├── 检查 input_doc 扩展名
   ├── .md → 跳过，直接进入步骤1
   ├── .pdf/.docx/.doc/.pptx/.ppt/.html → 调用 mineru-open-api extract：
   │   ├── mineru-open-api extract "{input_doc}" -o workspace/source/output/mineru_doc/{项目名称}_{hash}/ --model pipeline
   │   ├── 获取输出的 md 文件和 images/ 目录
   │   └── 将 md 路径作为后续步骤的输入
   └── 其他格式 → 报错退出

1. 解析输入参数
   ├── 验证 input_doc（或 mineru 输出的 md）是否存在
   └── 验证 output_dir 是否有效

2. 创建目录结构
   ├── 创建 output/{项目名称}/ 目录
   ├── 创建 03_requirements/ 目录
   └── 为每个章节创建章节目录和 images/ 子目录

3. 读取需求文档
   ├── 按行读取 md 文档
   ├── 识别章节结构（一级/二级标题）
   └── 提取段落内容

4. 章节遍历
   └── 对每个章节执行：
       ├── 识别功能模块
       ├── 在模块内识别功能需求单元
       ├── 提取验收标准
       ├── 关联图片（按同层级分组规则：图片向上查找关联到同级语义分组节点）
       ├── 将同一分组下的所有子项文本 + 所有图片作为一个完整需求单元
       ├── 生成 JSON 需求文档
       └── 【重要】验证每个需求的testable_elements完整性

5. 一致性校验（每个章节完成后必须执行）
   ├── 校验需求ID连续不间断
   ├── 校验模块requirement_count与实际数组长度一致
   ├── 校验章节total_requirements等于所有模块之和
   ├── 【重要】校验每个需求的testable_elements字段完整
   └── 如有不一致，报错并列出具体问题

7. 生成汇总
   ├── 生成各章节 requirements.json

   └── 输出执行报告
```

**可测试性验证（必须在步骤4执行）**：

```
对于每个识别出的需求单元：
1. 检查是否包含 trigger、inputs、action、expected_output
2. 如果缺少任一要素：
   - 尝试从detailed_description中推导
   - 如果无法推导，标记type为"非功能需求(NFR)"或跳过
3. 确保description格式为"[角色]可以[动作]..."
```

## 10. 错误处理

| 错误情况                     | 处理方式                                                         |
| ------------------------ | ------------------------------------------------------------ |
| 输入文档不存在                  | 报错退出，提示用户检查路径                                                |
| 文档格式异常                   | 跳过异常段落，记录日志                                                  |
| 图片文件缺失                   | 在 JSON 中标注 `image_missing: true`，继续处理                        |
| 图片复制失败                   | 记录警告，继续处理；不影响需求文档生成                                          |
| 验收标准无法提取                 | 从detailed\_description反向推导，仍无法推导则标注`acceptance_criteria: []` |
| 一致性校验失败                  | 报错退出，列出具体不一致项（如ID不连续、计数不匹配）                                  |
| **可测试性验证失败**             | 标记type为`非功能需求(NFR)`或跳过该需求，继续处理其他需求                           |
| **testable\_elements缺失** | 尝试从description推导，如无法推导则标记需求类型并添加note说明                       |

## 11. 需求识别示例

### 11.1 应提取的需求（功能性）

**示例1：计算公式类需求**

```markdown
{需求名称}

当满足 {条件A} 时，按以下公式计算 {结果变量}：
{结果变量} = ({参数A} × {系数1} - {参数B}) × Max({参数C} - {参数D}, 0) × {系数2}

其中：
- {系数1}：{系数说明}，默认值为 X%
- {系数2}：{系数说明}，默认值为 Y
- {边界条件} 时，该时段 {结果变量} 为 0
```

→ **提取为需求**："{需求名称}"，模块："{模块名}"

**示例2：条件判断类需求**

```markdown
{需求名称}

当 {条件表达式} 时，执行 {操作A}；否则执行 {操作B}。
```

→ **提取为需求**："{需求名称}"，模块："{模块名}"

### 11.2 不应提取的内容（背景/描述性）

**示例1：背景描述**

```markdown
{系统名称}是保障{业务领域}正常运行的核心环节。
系统支持多种{业务类型}和{规则类型}。
```

→ **不提取**：这是背景概述，不是具体功能操作

**示例2：业务前提**

```markdown
【{业务}基本规则】
1. {角色}需完成{前置条件}
2. {数据}需通过{校验}
3. {公式}需符合{规则文件}要求
```

→ **不提取**：这是业务规则说明，不是功能需求

**示例3：概念定义**

```markdown
{概念名称}是指当{主体}的{变量A}与{变量B}存在偏差时，
按约定的{规则类型}和{参数}进行{处理}的机制。
```

→ **不提取**：这是系统功能概述，不是可测试的需求点

**示例4：公式参数说明（无独立操作）**

```markdown
【公式参数】
- {系数1}：{说明}，默认值为 X
- {系数2}：{说明}，默认值为 Y
- {参数A}：{说明}
```

→ **不提取**：这是公式参数定义，不是功能操作。若某个参数需要独立的校验规则或条件分支，才单独提取。

### 11.3 判断流程

```
遇到一个段落
    │
    ▼
是否有明确动作动词？（新增/删除/修改/查询/展示/使用/导入/导出）
    │
    ├── 否 → 检查是否是业务规则/背景描述/概念定义
    │         └── 是 → 不提取为独立需求，跳过
    │
    └── 是 → 是否可设计具体测试用例？（有输入、输出、触发条件）
              │
              ├── 是 → 提取为需求
              └── 否 → 归类为业务规则，关联到最近的功能需求
```

### 11.4 不可测试需求的处理

**以下内容不应提取为独立需求：**

| 类型          | 处理方式           |
| ----------- | -------------- |
| 背景描述        | 跳过，不提取         |
| 概念/术语定义     | 跳过，不提取         |
| 业务前提/假设     | 跳过，不提取         |
| 概述性文字       | 跳过，不提取         |
| 字段属性说明（无操作） | 跳过，不提取         |
| 业务流程概述      | 关联到具体功能需求作为上下文 |
| 通用说明/注意事项   | 跳过，不提取         |

**如果某段文字描述的是"系统应该支持X"但没有具体操作步骤**：

1. 检查是否有具体的UC用例标题（如"UC-xxx"）
2. 如果有，按照UC用例提取为独立需求
3. 如果没有，将内容作为背景信息归并到最近的功能需求中

### 11.5 特殊需求类型标记

**对于无法完全跳过但又不完整的内容，使用type字段标记：**

```json
{
  "type": "非功能需求 (NFR)",
  "note": "仅包含业务规则，无法提取完整的测试要素"
}
```

可用的type值：

- `功能需求 (FR)` - 可直接测试的功能
- `非功能需求 (NFR)` - 性能、安全等无法直接测试的需求
- `业务规则 (BR)` - 业务规则说明，需关联到具体功能
- `数据需求 (DR)` - 数据字段定义，需关联到具体功能
