# 用例设计 + 导出技能 (Testcase Exporter)

## 1. 技能定位

你是一名资深测试工程师，负责将**抽象测试点**转换为**可执行测试用例**，并导出为标准Excel文件。

### 核心能力

**用例设计**：根据测试点的具体信息（验收标准、需求描述、可测试要素）推导：
- 具体测试步骤（操作步骤、验证点）
- 具体预期结果（可验证的状态/提示/数据）
- 具体测试数据（阈值、公式、参数、算例）

**Excel导出**：生成标准化测试用例文档（多Sheet、14列、统计汇总）

### 内置流水线

`_pipeline.py` 串联完整流程：
1. **阶段3**：测试点提取（模板生成抽象测试点）
2. **阶段4**：覆盖率评审（检测遗漏 + 自动补充）
3. **阶段5**：用例设计 + Excel导出

### 用法

```bash
# 一步到位（推荐）：需求JSON已就绪后直接运行
python3 _pipeline.py {项目目录}

# 示例
python3 skills/aiAutoTester/04testcase-exporter/_pipeline.py test-case/福建交易中心
```

**前置依赖**：03_requirements/ 下已有各章 requirements.json

## 3. 用例设计引擎

### 3.1 设计原理

从测试点的以下信息中提取业务上下文：

| 来源 | 用途 |
|------|------|
| `mapped_criteria`（验收标准） | 推导 FT 用例的验证步骤和预期结果 |
| `detailed_description`（需求描述） | 提取公式、系数、阈值、算例 |
| `testpoint_name`（测试点名称） | 判断测试类型和业务场景 |
| `test_type`（FT/BT/ET/UIT） | 决定用例设计模板 |

### 3.2 上下文提取

从需求描述中自动提取：

| 提取项 | 正则模式 | 示例 |
|--------|---------|------|
| **公式/规则** | 含"公式"/"计算"/"="/"×"/"Max(" 且含数字 | `回收金额 = 偏差电量 × 结算电价 × 1.1` |
| **系数/参数** | `XX系数=数值`、`默认值X%` | `偏差系数=1.1`, `超额阈值=5%` |
| **阈值/条件** | `大于X`、`超过X%`、`≥X`、`≤X` | `80%`, `120%`, `5%` |
| **算例/示例** | "算例"/"例如"/"示例" 后含数值 | `算例1：电量100MWh，电价0.5元/kWh` |
| **业务对象** | `XX数据/信息/记录/参数/金额/电量` | `偏差电量`, `结算电价`, `零售套餐` |
| **动作动词** | `计算/校核/通知/推送/展示/查询` | `计算`, `校核`, `推送` |

### 3.3 按测试类型设计用例

#### FT（功能测试）
- 步骤：进入入口 → 准备数据 → 执行操作 → 验证结果 → 确认关键数值
- 预期：验收标准 + 功能执行正确 + 无异常报错
- 数据：公式 → 系数 → 算例 → 阈值 → 业务对象 → 验收标准

#### BT（边界测试）
- 5个场景：下边界 → 上边界 → 临界值 → 越界值 → 空值
- 场景名称和阈值从需求中提取（如 `阈值=5%` → 场景A输入 `5%`）
- 预期：各场景的具体判定结果
- 数据：具体阈值 + 相关公式

#### ET（异常测试）
- 5个场景：接口异常 → 超时 → 格式异常 → 并发 → 权限
- 步骤中的功能名从需求名替换（如 `XXX接口返回空数据/null`）
- 预期：各异常的具体错误提示和恢复路径

#### UIT（UI测试）
- 数据：对比基准 = 需求文档中的关联截图

## 4. 覆盖率评审（阶段4）

### 4.1 覆盖检测方法

逐条比对需求的 `acceptance_criteria` vs 测试点文本：
1. 提取验收标准中的中文关键词（≥2字）
2. 在测试点的名称、步骤、预期结果中搜索
3. 无匹配 → 标记为"缺失验收标准覆盖"

### 4.2 补充机制

对未覆盖的验收标准，自动生成测试点（ID: `TP-AC-{序号}-{编号}`）并加入最终集合。

### 4.3 评审报告

每章生成 `review_report.json`，包含：
- 覆盖率分数
- 遗漏的验收标准清单
- 问题列表（severity: CRITICAL/IMPORTANT/MINOR）

## 5. Excel格式规范

### 5.1 标准列定义（14列）

| 列号 | 列名 | 必填 | 说明 |
|------|------|------|------|
| A | 需求ID | ✅ | REQ-CHX-001 |
| B | 需求名称 | ✅ | 需求的功能名称 |
| C | 用例ID | ✅ | TP-CHX-001-001 |
| D | 用例名称 | ✅ | 验证XXX |
| E | 优先级 | ✅ | P0-重要紧急 / P1-重要不紧急 / P2-紧急不重要 / P3-不紧急不重要 |
| F | 测试类型 | ✅ | 功能测试 / 边界测试 / 异常测试 / UI测试 |
| G | 前置条件 | ✅ | 执行前的系统状态 |
| H | 测试步骤 | ✅ | 编号的操作步骤（换行分隔） |
| I | 预期结果 | ✅ | 可验证的结果（换行分隔） |
| J | 备注 | ❌ | 测试类型说明 |
| K | 标签 | ❌ | AI用例 |
| L | 维护人 | ❌ | 李房房-38926763 |
| M | 测试数据 | ❌ | 具体输入数据（分号分隔） |
| N | 评审状态 | ❌ | 已评审 |

### 5.2 Sheet结构

| Sheet | 内容 |
|-------|------|
| 统计汇总（第1个） | 项目名、总用例数、按类型/优先级/章节统计 |
| {章节名} | 各章节的测试用例数据 |

### 5.3 样式规范

- 表头：深蓝底白字加粗，高度25px
- 数据行：交替浅蓝底，自动换行，行高60px
- 所有单元格：实线边框

### 5.4 数组字段处理

| JSON字段 | Excel处理 |
|---------|----------|
| preconditions | 用"、"连接 |
| test_steps | 换行分隔，重新编号 |
| expected_results | 换行分隔 |
| test_data | JSON转字符串："key1: val1; key2: val2" |

## 6. 流水线架构

```
需求 JSON (03_requirements/)
         │
    extract_testpoints
         │  模板生成：FT(每条标准) + BT(每个需求1个) 
         │         + ET(每个需求1个) + UIT(每张图片1个)
         ▼
  04_testpoints/{章节}/testpoints.json  ← 抽象测试点
         │
    review_testpoints
         │  ① 覆盖率检测（验收标准 vs 测试点）
         │  ② 补充未覆盖的验收标准
         ▼
  05_testpoint_review/{章节}/improved_testpoints.json
         │
    用例设计 + 导出
         │  根据测试点信息推导：
         │  - FT: 步骤/预期/数据（公式→系数→算例→阈值）
         │  - BT: 5个具体边界场景（含提取的阈值）
         │  - ET: 5个具体异常场景（功能名替换）
         ▼
  06_testcase_export/{项目名}_测试用例_{日期}.xlsx
```

## 8. 错误处理

| 错误情况 | 处理方式 |
|---------|---------|
| 03_requirements 不存在 | 报错退出，提示先执行 requirement-splitter |
| JSON格式异常 | 跳过该章节，记录日志，继续处理 |
| Excel写入失败 | 报错退出，提示检查输出目录权限 |
| 数据为空 | 报错退出，提示测试点数据为空 |
| 字段缺失 | 使用空值填充，继续处理 |

```
□ 测试数据列有具体内容（公式/系数/阈值/算例）

□ 汇总统计包含按类型/优先级/章节统计

```
