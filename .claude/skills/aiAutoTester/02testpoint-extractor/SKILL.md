# 测试点提取技能 (Testpoint Extractor)

## 1. 技能定位

本技能专注于**阶段3:测试点提取**,负责:

1. **项目类型识别**(API/跨境/金融/电力交易/普通),激活对应检查清单
2. 按章节读取需求 JSON 文档及关联图片
3. **遍历章节JSON中每一个需求,将每个需求描述和图片一起给大模型**提取这个需求的测试点
4. 每个需求的测试点提取完成之后,都要更新任务清单,一个一个需求点进行提取,可以并发调用子agent来完成任务
5. 接口覆盖度检查(API项目必做)
6. 跨境场景检查(跨境项目必做)
7. 业务流程完整性检查
8. 负面测试点规则("不支持"功能仅生成负面验证)
9. P0 冒烟测试识别(三问法)
10. 性能测试专项
11. 测试点标题正则校验(强制规范)
12. 覆盖度自检(<80% 告警 + 差距分析)

**前置依赖**:requirement-splitter 技能已成功执行并输出 JSON 需求文档和图片文件

---

## 1.5 项目类型识别(新增 - Step 0)

**目的**:识别项目类型,激活对应检查清单,确保测试覆盖无遗漏

| 项目类型 | 识别关键词 | 激活检查清单 |
|----------|------------|--------------|
| API/协议类 | OCPI、HTTP、接口、端点、协议、REST、API | 接口覆盖度检查 |
| 跨境/跨区域 | 跨境、香港、澳门、海外、多语言、多币种 | 跨境场景检查 |
| 支付/金融 | 支付、结算、对账、发票、税务、计费（排除"充电支付"、"停车支付"等场景支付） | 金融合规检查 |
| 电力交易 | 电力交易、电力市场、售电、购电、电价、负荷、用电、发电、配电、抄表、计量、偏差考核、中长期交易、现货交易、辅助服务、需求响应 | 电力交易场景检查 |
| 普通业务 | 无特殊关键词 | 基础检查 |

> **规则**:按关键词匹配,可同时激活多个类型(如 "香港+OCPI" = API + 跨境)
> 
> **排除词机制**:当文档同时包含排除词时,降低对应类型的置信度。例如"充电支付"、"停车缴费"等场景支付不属于金融类项目,不激活金融合规检查。

## 1.6 负面测试点规则(新增 - Step 0.5)

**目的**:正确理解"不支持"、"不涉及"功能,生成正确的负面测试点

| 需求描述 | 测试点类型 | 示例 |
|---------|-----------|------|
| 不支持 XX | 仅负面测试点 | 验证 XX 入口隐藏/置灰/不可用 |
| 不涉及 XX | 不生成测试点 | 完全不生成相关测试点 |
| 禁止 XX | 仅负面测试点 | 验证 XX 操作被禁止 |
| 无 XX 功能 | 不生成测试点 | 完全不生成相关测试点 |

> **强制规则**:对于明确说明"不支持"的功能,**禁止生成正面流程测试点**,只生成负面验证(验证入口隐藏/置灰/不可用)

### 5.1 测试点类型

| 类型     | 代码 | 说明                         |
| -------- | ---- | ---------------------------- |
| 功能测试 | FT   | 验证功能是否符合需求         |
| 边界测试 | BT   | 测试边界值条件               |
| 异常测试 | ET   | 测试异常情况和错误处理       |
| 性能测试 | PT   | 测试性能指标                 |
| UI测试   | UIT  | 测试界面交互(**基于图片**) |
| 安全测试 | ST   | 测试安全相关功能             |
| 合规测试 | CT   | 验证合规要求(金融/跨境等)  |
| 数据测试 | DT   | 验证数据准确性、精度、格式   |

### 5.2 优先级定义

| 优先级 | 说明                 | 适用场景                                     |
| ------ | -------------------- | -------------------------------------------- |
| P0-重要紧急 | 核心功能+必须立即验证 | 失败导致版本不可发布、阻塞后续测试、每次部署必跑 |
| P1-重要不紧急 | 重要功能但可稍后验证   | 影响主要流程、应该通过，但不阻塞发布或后续测试 |
| P2-紧急不重要 | 边缘功能但需尽快覆盖   | 不阻塞上线但需要快速验证的场景（如兼容性问题、偶发异常） |
| P3-不紧急不重要 | 低频场景、可选执行   | 边界条件、用户体验优化、低频使用场景           |

### 5.2.1 P0 冒烟测试识别(新增 - 三问法)

从已提取的测试点中识别 P0 冒烟测试（重要紧急）,**满足以下任一条件即可判定为 P0**:

1. ❓ 这条失败是否导致版本不可发布?→ 是 = P0（重要+紧急）
2. ❓ 这条失败是否阻塞所有后续测试?→ 是 = P0（重要+紧急）
3. ❓ 这条是否需要每次部署后必跑?→ 是 = P0（重要+紧急）

**P0 覆盖范围（重要紧急）**:
- 系统可用性(登录、服务健康检查)
- 核心业务闭环
- 安全底线(无效令牌拒绝、未授权访问拦截)
- 数据准确性(计费金额、关键计算结果)

**P1 识别参考（重要不紧急）**:
- 重要业务流程、常用功能场景
- 不影响发布但影响用户体验的关键功能
- 需要覆盖但可在 P0 之后执行的测试

**P2 识别参考（紧急不重要）**:
- 兼容性问题（特定浏览器/设备/分辨率）
- 偶发异常、边界条件快速验证
- 不影响核心功能但需尽快确认的场景

**P3 识别参考（不紧急不重要）**:
- 低频使用场景、边界值组合
- 用户体验优化、界面细节
- 可选执行的非关键测试

**输出要求**:单独生成 `00-P0_smoke_testpoints.md` 文件,包含执行检查表和自动化建议。

**P0 数量校验**:
- P0 = 0 → 警告:未识别 P0 测试点,建议补充核心冒烟测试
- P0 < 3 → 警告:P0 测试点过少,建议补充到 5-10 条
- P0 > 10% → 警告:P0 测试点过多,建议重新评估优先级

### 5.4 接口覆盖度检查(新增 - API 项目必做)

**触发条件**:项目类型识别为 API/协议类时强制执行

1. **梳理接口清单**
   ```
   - 从需求 JSON 中提取所有接口/端点列表
   - 标记每个接口的 HTTP 方法、路径、必填字段、可选字段
   - 识别接口间依赖关系
   ```

2. **生成接口覆盖度检查表**
   ```markdown
   | 接口 | 方法 | 覆盖状态 | 测试点数 |
   |------|------|----------|----------|
   | /tokens | GET | ✅ | 3 |
   | /tokens | PUT | ✅ | 2 |
   | /sessions | POST | ❌ | 0 |
   ```

3. **识别遗漏接口**
   - 协议要求但需求文档未提及的接口 → 标记为"隐性需求"
   - 补充对应测试点

```
[ ] 辅助服务市场（调频、调峰、备用容量）
[ ] 需求响应（邀约、执行、补贴核算）
[ ] 结算与对账（电费核算、偏差结算、补贴结算）
[ ] 市场主体注册与准入（发电企业、售电公司、电力用户资质审核）
[ ] 数据上报与监管合规（向电力交易中心/调度机构数据报送）
```

每个勾选框至少对应 1 个测试点。

> **排除词机制**:当文档仅涉及"充电桩计费"、"停车缴费"等场景用电计费时,不激活电力交易检查清单。

---

### 5.6 业务流程完整性检查(新增)

1. **梳理端到端业务流**(从需求 JSON 中提取关键流程节点和分支)
2. **识别所有分支流程**:正常流程、异常流程、取消流程、退款流程
3. **识别所有子流程**:预约、预授权、通知、对账等
4. **确保每个流程节点都有对应测试点覆盖**

---

### 5.7 性能测试专项(新增)

**性能测试点必须包含的要素**:
1. **并发数**:明确并发用户数/请求数
2. **响应时间要求**:明确阈值(如 P95<500ms)
3. **成功率要求**:明确阈值(如 >99.9%)
4. **持续时间**:压测持续时间

**性能测试点模板**:
```
[P2-紧急不重要][性能] 验证 [场景],[并发数/QPS] 持续 [时间],P95 响应时间<[阈值],成功率>[阈值]%
```

**好示例**:
- ✅ `[P2-紧急不重要][性能] 验证 100 并发请求,持续 10 分钟,P95 响应时间<500ms,成功率>99.9%`

**差示例**(必须避免):
- ❌ `[P2-紧急不重要] 验证系统性能`(无具体指标)
- ❌ `[P2-紧急不重要] 验证并发处理`(无并发数/阈值)

---

### 5.8 测试点标题规范校验(新增 - 强制)

**标题模板**:
```
[P0-重要紧急/P1-重要不紧急/P2-紧急不重要/P3-不紧急不重要] 验证 + [具体场景/页面] + [关键条件/输入] + [预期结果关键词]
```

**标题要素**(必须全部包含,缺一不可):
1. **具体场景**:哪个页面/哪个功能/哪个接口
2. **关键条件**:什么前提条件/什么输入数据
3. **验证点**:验证什么结果(动词:显示、展示、处理、校验、锁定、禁止、扣费、同步、查询)

**正则表达式校验**:
```regex
^\[P[0-3]-(重要紧急|重要不紧急|紧急不重要|不紧急不重要)\] 验证.+ (页面 | 接口 | 场景 | 功能 | 服务).+ (显示 | 展示 | 处理 | 校验 | 锁定 | 禁止 | 扣费 | 同步 | 查询).+$
```

**自检规则**:
1. 标题长度 ≥15 字
2. 包含场景类型关键词(页面/接口/场景/功能/服务)
3. 包含预期结果动词
4. 包含具体条件
5. 匹配正则表达式

**不匹配处理**:丢弃该标题,重新生成(最多重试 2 次;若仍不匹配,保留原标题并在 notes 中标记"标题需人工审核")

---

### 5.9 覆盖度自检(新增)

生成测试点后必须执行:
```
接口覆盖度 = 已覆盖接口数 / 总接口数 × 100%
流程覆盖度 = 已覆盖流程节点数 / 总节点数 × 100%
跨境覆盖度 = 已覆盖跨境场景数 / 应覆盖跨境场景数 × 100%

if 覆盖度 < 80%:
    生成 test-case/gap-analysis.md 差距分析报告
    输出告警:"覆盖度仅 XX%,建议补充 XX 模块"
else:
    输出 "覆盖度检查通过"
```

**差距分析报告模板**:
```markdown
# 测试点差距分析报告

## 覆盖度统计
- 接口覆盖度:XX%
- 流程覆盖度:XX%
- 跨境场景覆盖度:XX%

## 遗漏功能点
| 模块 | 遗漏点 | 优先级 | 建议测试点数 |
|------|--------|--------|-------------|
| 预约 | 预约超时释放 | P2-紧急不重要 | 2 |

## 建议补充测试点清单
(列出具体建议补充的测试点)
```

## 8. 执行流程

```
0. 项目类型识别（新增）
   ├── 扫描需求 JSON 关键词
   ├── 识别项目类型（API/跨境/金融/普通）
   ├── 激活对应检查清单
   └── 负面测试点规则初始化

1. 解析输入参数
   ├── 验证 requirements_json 是否存在
   ├── 验证 output_dir 是否有效
   └── 验证 images/ 目录是否存在

2. 读取需求文档
   ├── 解析 requirements.json
   ├── 解析 requirements_detail.json（如有）
   └── 构建需求结构树

3. 图片加载（使用 image 工具读取）
   └── 对每个需求：
       ├── 读取 related_images 中的图片路径
       └── 调用 image 工具分析图片内容，转为文字描述后与需求描述组合

4. 遍历需求并提取测试点
   └── 对每个需求执行：
       ├── 判断是否为“不支持/不涉及”功能 → 负面测试点
       ├── 组装“需求描述+图片”联合输入
       ├── 调用大模型提取测试点
       ├── 解析大模型返回的JSON
       ├── 标题规范校验（正则 + 四要素）
       └── 保存测试点 JSON

5. 专项提取（根据项目类型）
   ├── [API项目] 接口覆盖度检查：梳理接口清单 → 生成覆盖度表 → 补充遗漏
   ├── [跨境项目] 跨境场景检查：9 项检查清单 → 补充对应测试点
   ├── [所有项目] 业务流程完整性检查：绘制流程图 → 识别分支/子流程 → 补充覆盖
   ├── [所有项目] 性能测试专项：核心接口 → 性能测试点
   └── [所有项目] P0 冒烟测试识别：三问法 → 生成 00-P0_smoke_testpoints.md

6. 覆盖度自检（新增）
   ├── 计算接口/流程/跨境覆盖度
   ├── 覆盖度 ≥ 80% → 输出“覆盖度检查通过”
   └── 覆盖度 < 80% → 生成 gap-analysis.md + 告警

7. 生成汇总
   ├── 生成各章节 testpoints.json
   ├── 生成全局汇总 testpoints_summary.json
   ├── 生成 README.md（索引）
   ├── 更新任务清单
   └── 输出执行报告（含覆盖度统计）
```

## 9. 错误处理

| 错误情况       | 处理方式                                    |
| -------------- | ------------------------------------------- |
| 需求JSON不存在 | 报错退出,提示先执行 requirement-splitter   |
| 图片文件缺失   | 继续处理,但测试点 notes 中标注"图片缺失"   |
| JSON格式异常   | 先尝试从大模型回复中提取 ```json``` 代码块;若仍无法解析,跳过该需求,记录日志,继续处理其他需求 |
| JSON 结构校验失败(缺少必填字段) | 记录日志并标注缺失字段,自动补全默认值后继续;若缺失超过3个必填字段,跳过该需求 |
| 验收标准为空   | 生成至少1个通用功能测试点                   |
| 大模型调用失败 | 间隔 5 秒后重试,最多重试 3 次;仍失败则使用规则引擎生成基础测试点(基于需求标题和描述生成 FT+BT) |
| 图片分析失败   | 继续处理,但测试点 notes 中标注"图片内容未解析,建议人工补充" |

## 11. 四个 Skill 的协作约定

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Skill 协作流程                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. requirement-splitter (解析 + 拆分)                              │
│     文档 PDF/Word/PPT/md → MinerU 转 md → 需求拆分 → JSON需求文档   │
│     输出:03_requirements/                                          │
│     ├── xx_chapter/                                                 │
│     │   ├── requirements.json                                      │
│     │   ├── requirements_detail.json                               │
│     │   └── images/ ← 关联图片                                     │
│     └── requirements_summary.json                                   │
│                                                                     │
│  2. testpoint-extractor (提取)                              │
│     读取:03_requirements/ + images/                                │
│     提取:项目类型识别 → 接口覆盖 → 跨境场景 → P0冒烟 → 性能专项     │
│     校验:标题规范 → 覆盖度自检 → 负面测试点检查                      │
│     输出:04_testpoints/ + gap-analysis.md              │
│                                                                     │
│  3. testpoint-reviewer (评审)                                       │
│     读取:04_testpoints/ + 03_requirements/                         │
│     评审:四维评审 → 去重合并 → 标题校验 → 负面检查 → P0专项         │
│     输出:05_testpoint_review/ + improved_testpoints.json           │
│                                                                     │
│  4. testcase-exporter (导出)                                        │
│     读取:05_testpoint_review/improved_testpoints.json              │

│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 协作约定

| Skill                | 输出物                 | 后续 Skill 使用              |
| -------------------- | ---------------------- | ---------------------------- |
