# 测试点评审技能 (Testpoint Reviewer)

## 1. 技能定位

你是一名资深软件测试评审工程师（Quality Assurance Lead）。本技能专注于**阶段4：测试点质量评审**，负责：

1. 对比测试点与原始需求，验证覆盖完整性
2. 检查测试点的正确性、准确性、可执行性
3. 识别测试遗漏、描述模糊、优先级不当等问题
4. **测试点去重检查与合并**（新增）
5. **测试点标题规范校验**（新增）
6. **负面测试点合规检查**（新增）
7. **P0 冒烟测试专项评审**（新增）
8. **覆盖度差距分析**（新增）
9. 输出结构化评审报告 + 改进后的测试点

**前置依赖**：testpoint-extractor 技能已成功执行并输出测试点 JSON

## 3. 评审维度

### 3.1 四大评审维度

| 维度   | 代码 | 说明            | 检查要点                  |
| ---- | -- | ------------- | --------------------- |
| 完整性  | C  | Coverage      | 是否覆盖所有验收标准？是否有遗漏的功能点？ |
| 准确性  | A  | Accuracy      | 测试点是否准确反映需求？边界条件是否正确？ |
| 可执行性 | E  | Executability | 测试步骤是否清晰可执行？预期结果是否明确？ |
| 一致性  | I  | Consistency   | 与需求的描述是否一致？优先级是否合理？   |

### 3.2 问题严重等级

| 等级 | 代码        | 说明                 | 处理方式 |
| -- | --------- | ------------------ | ---- |
| 严重 | CRITICAL  | 测试点缺失或错误，可能导致功能不可用 | 必须修复 |
| 重要 | IMPORTANT | 测试覆盖不完整或描述不清       | 建议修复 |
| 一般 | MINOR     | 优化建议，不影响功能验证       | 可选修复 |

## 4. 评审检查项

### 4.1 完整性检查（Coverage）

```
□ 每个验收标准是否都有对应的测试点？
□ 是否缺少必要的功能测试点？
□ 边界条件是否充分覆盖？（如最大/最小值、空值、临界值）
□ 异常场景是否覆盖？（如错误输入、系统异常）
□ 特殊路径是否覆盖？（如改期、退票、取消）
```

### 4.2 准确性检查（Accuracy）

```
□ 测试点描述是否与需求描述一致？
□ 测试数据类型是否与需求匹配？
□ 预期结果是否符合业务逻辑？
□ 优先级设置是否与需求重要性匹配？
```

### 4.3 可执行性检查（Executability）

```
□ 测试步骤是否清晰、完整？
□ 步骤顺序是否合理？
□ 是否有必要的预置条件？
□ 预期结果是否可验证？
□ 测试数据是否具体可操作？
```

### 4.4 一致性检查（Consistency）

```
□ 测试点ID是否唯一且连续？
□ 测试类型分类是否正确？
□ 同一需求的测试点风格是否统一？
□ 映射的验收标准是否准确？
```

## 8. 测试点去重机制（新增）

**目的**：避免重复测试点，减少冗余

**重复测试点识别规则**（满足任一即视为重复）：

1. **相同验证点**在不同测试点中重复出现
   - 提取标题中的验证点关键词（动词 + 宾语）
   - 如相同关键词出现≥2 次 → 标记为疑似重复

2. **前置条件相同**且**预期结果相同**
   - 对比两个测试点的前置条件和预期结果字段
   - 如相似度>80% → 标记为疑似重复

3. **基础功能**在多个场景中重复验证
   - 识别基础功能关键词
   - 如基础功能在多个测试点中出现 → 标记为疑似重复

**去重策略**：

| 场景 | 策略 | 示例 |
|------|------|------|
| 相同验证点，不同条件 | 合并为参数化测试点 | 空闲/故障/占用 → 桩状态异常 |
| 相同验证点，不同优先级 | 保留最高优先级 | P1-重要不紧急+P2-紧急不重要 → 保留 P1-重要不紧急 |
| 相同验证点，不同详细程度 | 保留最完整测试点 | 3 步 vs 5 步 → 保留 5 步 |

**《测试点去重报告》模板**：
```markdown
## 测试点去重报告

### 合并的测试点
| 原编号 | 合并后编号 | 合并原因 |
|--------|-----------|----------|
| TP-001, TP-002, TP-003 | TP-001 | 桩状态校验重复 |

### 最终测试点统计
- 评审前：15 条
- 合并：5 条 → 3 条
- 评审后：13 条
```

## 9. 测试点标题规范校验（新增）

**标题模板**：
```
[P 优先级] 验证 + [具体场景/页面] + [关键条件/输入] + [预期结果关键词]
```

**正则表达式校验**：
```regex
^\[P[0-3]\] 验证.+ (页面 | 接口 | 场景 | 功能 | 服务).+ (显示 | 展示 | 处理 | 校验 | 锁定 | 禁止 | 扣费 | 同步 | 查询).+$
```

**检查规则**：
| 检查项 | 规则 | 不满足处理 |
|--------|------|-----------|
| 标题长度 | ≥15 字 | 标记问题，建议扩展 |
| 场景类型 | 包含页面/接口/场景/功能/服务 | 标记问题，要求补充 |
| 预期动词 | 包含显示/展示/处理/校验/锁定/禁止/扣费/同步/查询 | 标记问题，要求补充 |
| 关键条件 | 包含具体条件 | 标记问题，要求补充 |
| 正则匹配 | 匹配上述正则表达式 | 标记问题，要求修正 |

**差标题示例**（发现即标记为问题）：
- ❌ `[P1-重要不紧急] 验证价格展示` → 无具体场景、无关键条件
- ❌ `[P1-重要不紧急] 验证充电流程` → 无具体环节、无验证点
- ❌ `[P1-重要不紧急] 验证系统功能` → 无意义

## 10. 负面测试点合规检查（新增）

**目的**：确保"不支持"功能只生成负面验证，无正面流程测试点

**检查规则**：
- [ ] 所有"不支持"功能 → 仅生成负面测试点（验证入口隐藏/置灰）
- [ ] 所有"不支持"功能 → 无正面流程测试点
- [ ] 所有"不涉及"功能 → 无测试点生成
- [ ] 负面测试点的预期结果包含明确的验证标准（隐藏/置灰/提示信息）

**违规处理**：将正面测试点标记为 CRITICAL 问题，建议删除或转为负面测试点

---

## 11. P0 冒烟测试专项评审（新增）

**检查项**：
- [ ] 已识别 P0 测试点（5-10 条）
- [ ] P0 测试点占比 ≤10%
- [ ] P0 测试点覆盖核心业务闭环
- [ ] P0 测试点覆盖安全底线
- [ ] P0 测试点有执行检查表
- [ ] P0 测试点有自动化建议

**P0 = 0** → 标记为 CRITICAL：未识别 P0 测试点，建议补充核心冒烟测试
**P0 < 3** → 标记为 IMPORTANT：P0 过少，建议补充到 5-10 条
**P0 > 10%** → 标记为 IMPORTANT：P0 过多，建议重新评估优先级

## 13. 评审执行流程

```
1. 解析输入参数
   ├── 验证 testpoints_json 是否存在
   ├── 验证 requirements_json 是否存在
   └── 确定输出目录

2. 读取文件
   ├── 解析 testpoints.json
   ├── 解析 requirements.json
   └── 构建需求-测试点映射表

3. 执行评审（按维度）
   ├── 完整性评审：对照验收标准检查覆盖率
   ├── 准确性评审：检查描述一致性
   ├── 可执行性评审：检查步骤清晰度
   ├── 一致性评审：检查格式和分类
   ├── 去重检查：识别并合并重复测试点
   ├── 标题校验：验证标题格式规范性
   ├── 负面检查：验证"不支持"功能仅负面测试
   └── P0 专项：验证 P0 覆盖是否充分

4. 生成改进后的测试点（可选）
   ├── 根据评审问题自动改进测试点
   ├── 合并重复测试点
   ├── 修正不合规标题
   ├── 补充缺失的 P0 测试点
   └── 删除违规的正面测试点

5. 生成评审报告
   ├── 汇总问题
   ├── 计算评分
   ├── 输出《测试点去重报告》
   ├── 输出《标题检查表》
   ├── 生成差距分析报告（覆盖度<80%时）
   └── 保存报告

6. 输出结果
   ├── 章节评审报告
   ├── 全局评审汇总
   ├── 改进后的测试点（improved_testpoints.json）
   └── 更新任务清单
```
