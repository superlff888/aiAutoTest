# 量化指标参考表 · 数字写法对照手册

> **规则**：所有数字必须是你能解释来源的真数据；做不到也宁可不写，不要编造。
> 下表为合理区间参考，避免出现"召回率提升 200%"这种数学不可能的数字。

## 1. 性能类（吞吐/延迟/资源）

| 指标 | 合理区间 | 写法示例 |
|------|---------|----------|
| LLM 接口 P95 延迟 | 500ms - 3s | "首字延迟 P95 850ms，全文 P95 2.3s" |
| QPS / 吞吐 | 单实例 10-100 QPS | "vLLM 部署单卡 35 QPS" |
| Token 成本下降 | 20% - 60% | "通过 Prompt 压缩 + 缓存，单次成本降 42%" |
| 缓存命中率 | 20% - 50% | "Redis 语义缓存命中率 38%" |
| 检索召回 | 改进幅度 5-30 pp | "BGE-Reranker 加入后召回 72% → 89%" |
| GPU 显存占用 | 下降 30-70% | "INT4 量化后显存从 24GB → 8GB" |

## 2. 质量类（准确/覆盖/通过率）

| 指标 | 合理区间 | 写法示例 |
|------|---------|----------|
| 用例覆盖率 | 提升 10-40 pp | "等价类覆盖 65% → 95%" |
| Agent 任务成功率 | 60% - 95% | "复杂任务端到端成功率 78%" |
| 幻觉检测召回 | 70% - 92% | "幻觉检测召回 85%，误报率 < 8%" |
| Prompt 通过率 | 改进幅度 15-40 pp | "Prompt 评估通过率 62% → 91%" |
| Bug 漏出率下降 | 30% - 70% | "生产 Bug 漏出从 8.5‰ 降到 3.2‰" |
| 准确率 | 提升 5-25 pp | "意图识别准确率 81% → 93%" |

## 3. 效率类（耗时/工时/规模）

| 指标 | 合理区间 | 写法示例 |
|------|---------|----------|
| 人工 vs Agent 耗时 | 50% - 95% 下降 | "用例编写 30min → 2min（-93%）" |
| 评估周期 | 小时 → 分钟 | "回归评估 4h → 12min" |
| 测试数据规模 | 几百 - 几万条 | "构建 5000 条业务对话集" |

| 自动化覆盖率 | 提升 20-50 pp | "回归自动化率 45% → 88%" |

## 4. 业务类（用户/调用/收入）

| 指标 | 合理区间 | 写法示例 |
|------|---------|----------|
| 月调用量 | 万级 - 百万级 | "月调用 12 万次" |
| 用户数 | 数百 - 数万 | "服务内部用户 3000+" |
| 覆盖业务场景 | 3 - 20 个 | "覆盖客服 / 销售 / 培训 3 大场景" |
| 团队推广 | 2 - 10 个团队 | "在 3 个产品线推广" |
| 客诉减少 | 20% - 70% | "AI 客服上线后客诉下降 45%" |
| 年节约成本 | 几万 - 几百万 | "年节约人力成本 80 万元" |

## 5. 写数字的 5 条铁律

1. **优先用绝对值 + 百分比双指标**：不要只写 "提升 30%"，要写 "从 65% 提升到 85%（+20 pp）"
2. **pp 和 % 别混淆**：65% → 85% 是 +20 pp，不是 +30%
3. **避免数学不可能**：召回率/准确率提升不可能超过 100 pp，节省时间不可能超过 100%
4. **大数要可信**：月调用 10 亿次需要团队 + 资源背书，个人项目慎写
5. **数字越精确越可信**：写 "+23 pp" 比 "约 20%" 更让人相信你真的测过

## 6. 没有真数据怎么办？

- ❌ 不要编造（面试一问就穿帮）
- ✅ 用**合理估算**：基于业务规模合理推算（"按团队 N 人 × 单人月编写 M 条计算"）
- ✅ 用**对比基线**：没绝对数就用相对（"对比团队平均水平高 30%"）
- ✅ 用**结果指标替代过程指标**：没有性能数据，就写业务结果（"客户满意度 4.6/5"）
