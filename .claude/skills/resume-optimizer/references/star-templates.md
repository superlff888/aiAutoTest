# STAR 项目模板 · 万能写作骨架

> **STAR 法则**：Situation（情境） → Task（任务） → Action（行动） → Result（结果）
> **写作铁律**（鱼皮版）：**结论先行 + 量化数字 + 技术细节 + 业务影响**

## 1. 通用骨架（套用任何项目）

```
【项目名称】（年月-年月）  独立完成 / 核心开发（N 人）

【项目背景 - Situation】
一句话说清业务痛点 + 技术挑战。
示例：公司 N 个产品文档分散，员工平均查询 X 分钟仍找不到答案，
     现有关键字搜索召回率不足 50%。

【技术方案 - Action】
1. 【架构层】采用 XX 框架/范式 + 关键技术决策
   示例：基于 LangGraph 构建 StateGraph 工作流，含 N 个节点 + M 条边
2. 【核心模块】解决 N 个关键问题：① ... ② ... ③ ...
   示例：① 文档分块策略：层次分块 + 语义分块，平均块长 512 token
         ② 检索策略：BGE-large 向量召回 Top-20 + bge-reranker 重排 Top-3
         ③ 生成策略：tree_summarize 模式，温度 0.3，避免幻觉
3. 【创新点】对比 baseline 的差异化设计
   示例：引入覆盖率反馈环，未达 100% 自动触发补充生成（参考 ReAct 思想）

【最终结果 - Result】
- 量化指标 1：召回率从 X% 提升到 Y%（绝对提升 Z pp）
- 量化指标 2：响应延迟 P95 < N 秒（QPS = M）
- 业务影响：用户查询耗时下降 X%，覆盖 Y 个业务场景，月调用 Z 万次
- 工程产出：开源代码 N 行 / 申请专利 N 项 / 内部技术分享 N 次
```

## 2. RAG 项目示例

```
【企业知识库智能问答系统】（2025.09 - 2026.01）  独立完成

背景：销售/客服每天处理 200+ 重复咨询，平均响应 8 分钟，
     文档分散在 Confluence/钉钉/SharePoint 三个系统。

技术方案：
1. 基于 LightRAG 搭建 RAG 服务（端口 9621），整合向量+图双引擎
2. 文档管线：SimpleDirectoryReader → Qwen3-Embedding-8B → Chroma
3. 检索优化：top_k=3、tree_summarize 合成、加入图检索补充结构化关系
4. 工程化：FastAPI 接口 + Redis 缓存（命中率 35%）+ Prometheus 监控

结果：
- 首版召回率 65% → 优化后 88%（+23 pp）
- P95 延迟 1.2s，月调用 12 万次
- 客服平均响应时间 8min → 1.5min（-81%）
- 已替代 60% 一线重复咨询
```

## 3. Agent 项目示例

```
【AI 用例生成 Agent 工作流】（2025.10 - 2026.03）  核心开发（2 人）

背景：测试团队 N 人，每个版本人工编写用例 500+ 条，平均耗时 X 人天，
     覆盖率难以量化，存在大量重复劳动。

技术方案：
1. 基于 LangGraph StateGraph 编排 5 节点工作流：
   用例生成 → 用例评审（并行 Send）→ 覆盖率检查 → 补充生成（循环）→ 存储
2. 状态管理：TypedDict + Annotated[List, operator.add] 实现增量累加
3. 评审并行化：使用 Send API 将 N 条用例分发到 N 个并行节点评审
4. 闭环优化：覆盖率 < 100% 自动回到补充生成节点（最多 3 轮）
5. 结构化输出：Pydantic 模型 GenerateCase / CaseList 保证 LLM 输出可解析
6. 数据持久化：MySQL 存储用例 + 评审记录，支持版本追溯

结果：
- 单个需求点用例生成耗时：人工 30 min → Agent 2 min（-93%）
- 覆盖率从主观估计变为量化数据：平均 95%+
- 节省测试团队约 N 人天/月，年节省人力成本 X 万元
- 沉淀 200+ 通用 Prompt 模板，已在 N 个项目复用
```

## 4. AI 测试项目示例

```
【LLM 应用自动化评估平台】（2025.06 - 2026.02）  独立完成

背景：业务侧上线 5 个 LLM 应用，缺乏统一评估手段，
     人工评测 Prompt 改动效果需 2 天，无法支持快速迭代。

技术方案：
1. 集成 promptfoo + Giskard，构建 LLM 评估流水线
2. 评估维度：准确性（精确匹配/语义相似度）、鲁棒性（红队 Jailbreak 测试）、
            性能（P95 延迟、Token 消耗）、安全（毒性 / PII 检测）
3. 测试数据集：自建 500 条业务问答 + 引入 RAGAS 标准评估集
4. CI/CD 集成：Prompt 修改触发 GitLab Runner 自动评估，结果回填 MR
5. 可观测性：LangFuse 记录全链路 trace，按业务维度统计准召

结果：
- Prompt 评估周期：2 天 → 15 分钟（-99%）
- 上线前发现 23 个潜在幻觉问题，避免 8 起客诉
- 拦截 2 起 Prompt 注入攻击
- 形成《LLM 应用测试规范 V1.0》在 3 个团队推广
```

## 5. 写作技巧（来自鱼皮风格）

- **动词开头**：每行第一个词必须是动词（设计/实现/优化/搭建/推动）
- **数字优先**：每个 bullet 至少有 1 个数字
- **避免堆砌**：单条不超过 2 行，超过就拆分
- **结论先行**：先写结果再写方法，HR 5 秒决定要不要看你
- **STAR 不必平均**：Result 占 40%，Action 占 40%，Situation 占 20%
