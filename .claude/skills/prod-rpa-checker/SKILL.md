# prod-rpa-checker 技能说明

## 功能概述

校验各交易中心数据是否采集到最新且日期无中断：

1. **最新数据时间验证**：根据 JSON 配置，验证数据是否采集到最新（如 D-1、D-2）
2. **日期连续性验证**：验证指定时间范围内每一天的数据都存在

## 数据中心配置

- **JSON配置**：[数据中心类型定义.json](doc/数据中心类型定义.json)
- **MD文档**：[数据中心类型定义.md](doc/数据中心类型定义.md)

配置结构：

```json
{
  "广东": {
    "trade_center_id": 1,
    "日前节点电价": "D",
    "实时节点电价": "D-2",
    ...
  },
  ...
}
```

## 参数转换规则

D加减时间转换为 offset_days：

- D → 0
- D-1 → 1
- D-2 → 2
- D-3 → 3
- D+1 → -1
- D+2 → -2

多个时间值（如 D-1/D-2）：优先查询 D-1，查询不到再查 D-2

## 校验流程

### 1. 最新数据时间验证

对于每个交易中心的数据类型：

1. 读取 JSON 配置中的最新数据时间
2. 如果为 "暂未接入"，跳过
3. 如果有多个时间值（如 D-1/D-2），先查 D-1
4. 执行 SQL，查询最大日期
5. 与预期最新数据时间对比：
   - 预期 D，查询到的最大日期应为今天
   - 预期 D-1，查询到的最大日期应为昨天
   - 如不符，断言失败并说明原因

### 2. 日期连续性验证

对于指定时间范围（如月初至今）：

1. 生成日期序列
2. 查询数据库中的所有日期
3. 对比找出缺失的日期
4. 如有缺失，断言失败并列出缺失日期

### 3. 数据量合理性校验

对每个数据类型（除网侧预测/网侧实际外），检查数据量是否合理：

- **计算预期**：以数据最大日期为基准，计算从月初到该日期的天数 N
- **阈值判断**：
  - 数据量 < N × 80% → ❌ 数据不足（可能有漏采）
  - 数据量 > N × 120% → ❌ 数据异常（可能有重复或脏数据）
  - 80% ≤ 数据量 / N ≤ 120% → ✅ 合理

## 执行示例

用户输入：

```
检查广东的日前节点电价数据
```

技能执行：

1. 读取 JSON 配置，广东的日前节点电价最新数据时间为 D
2. 获取 trade_center_id = 1
3. offset_days = 0
4. 读取 [日前节点电价.sql](sqls/日前节点电价.sql)
5. 替换参数执行 SQL
6. 验证最大日期是否为今天
7. 返回验证结果

## 降级查询与日期连续性验证

**核心思想**：先找到"最近一次有数据的日子"，然后验证从月初到那天的数据是否完整。

**降级查询流程**（以安徽代理用户用电明细 [D-4, D-5, D-6] 为例）：

```
第一优先：查 D-4
├─ 有数据 → 验证 月初 ~ D-4 是否连续
├─ 无数据 → 降级查 D-5
│   ├─ 有数据 → 验证 月初 ~ D-5 是否连续
│   ├─ 无数据 → 降级查 D-6
│   │   ├─ 有数据 → 验证 月初 ~ D-6 是否连续
│   │   └─ 无数据 → 报告 月初 ~ D-6 缺失几天
```

**关键规则**：

- 不管最终停在哪一级，连续性验证范围 = 月初 ~ 有数据的日期
- 所有 offset 都无数据时，缺失日期范围 = 月初 ~ 最后一个 fallback 日期（不是今天）
- 连续性验证的结束日期 = offset 对应的预期日期，不是全局的今天

## 注意事项

- 暂未接入的数据类型会被跳过，不进行校验
- 多时间值的数据类型，优先查询较新时间，查询不到再查较旧时间
- 断言失败时需明确指出预期与实际的差异
- 日期缺失时需列出具体的缺失日期
