# prod-rpa-checker 技能说明

## 功能概述

校验各交易中心数据是否采集到最新且日期无中断：

1. **最新数据时间验证**：根据 JSON 配置，验证数据是否采集到最新（如 D-1、D-2）
2. **日期连续性验证**：以最新数据日期为终点，往前推10天，验证每一天数据都存在

## 数据中心配置

技能下挂两份 JSON，分别给**上午任务**和**下午任务**用：

| 文件 | 调度任务 | 用途 |
|------|---------|------|
| [数据中心类型定义am.json](doc/数据中心类型定义am.json) | RPA-Data-Check-Morning（08:35） | 上午任务用 |
| [数据中心类型定义pm.json](doc/数据中心类型定义pm.json) | RPA-Data-Check-Afternoon（17:15） | 下午任务用 |

> ⚠️ 两份 JSON 内容**不同**（pm.json 整体后移 1 天）。改其中任何一个前，**先确认改的是哪个文件**。

配置结构：

```json
{
  "广东": {
    "trade_center_id": 1,
    "vpp_id": "1",
    "日前节点电价": "D",
    "实时节点电价": ["D-1", "D-2"],
    "网侧预测": "D+1",
    ...
    "中长期持仓": "D",
    "未接入的数据类型": { "最新数据时间": "—", "备注": "暂未接入" }
  },
  ...
}
```

字段说明：

- `trade_center_id`：数据库主键，用于 SQL `{{trade_center_id}}` 占位符
- `vpp_id`：VPP 标识符
- 时间值字段：字符串 `D±N`，或数组 `["D-N", ...]`（数组用于降级查询）
- `暂未接入`：用对象形式标记，校验时跳过该数据类型

## SQL模板

SQL文件位于 `sqls/` 目录，使用占位符：

- `{{trade_center_id}}` - 交易中心ID
- `{{offset_days}}` - D加减天数（D-1=1, D-2=2, D+1=-1, D=0）

| SQL文件              | 对应数据类型     |
| -------------------- | ---------------- |
| 代理用户用电明细.sql | 代理用户用电明细 |
| 代理用户用电总计.sql | 代理用户用电总计 |
| 日前成交电量.sql     | 日前成交电量     |
| 实时节点电价.sql     | 实时节点电价     |
| 实时结算电价.sql     | 实时结算电价     |
| 网侧实际.sql         | 网侧实际         |
| 网侧预测.sql         | 网侧预测         |
| 日前节点电价.sql     | 日前节点电价     |
| 日前结算电价.sql     | 日前结算电价     |

## 参数转换规则

D加减时间转换为 offset_days：

- D → 0
- D-1 → 1
- D-2 → 2
- D-3 → 3
- D+1 → -1
- D+2 → -2

多个时间值（如 D-1/D-2）：优先查询 D-1，以下两种情况均降级到 D-2：
- 完全无数据
- 有数据但最大日期与预期不匹配

## 校验流程

### 1. 最新数据时间验证

对于每个交易中心的数据类型：

1. 读取 JSON 配置中的最新数据时间
2. 如果为 "暂未接入"，跳过
3. 如果有多个时间值（如 D-1/D-2），先查 D-1
4. 执行 SQL，查询最大日期
5. 与预期最新数据时间对比：
   - 预期 D，查询到的最大日期应为今天
   - 预期 D-1，查询到的最大日期应为昨天
   - 如果**无数据**或**最大日期与预期不符**，降级到下一个offset（如 D-2）
   - 所有 offset 都失败时，返回最终错误信息

### 2. 日期连续性验证

匹配成功后：
1. 以匹配日期为终点，往前推9天作为起点（共10天窗口）
2. 查询该10天窗口内的所有日期
3. 对比找出缺失的日期
4. 如有缺失，断言失败并列出缺失日期

### 3. 数据量合理性校验

对指定数据类型（网侧预测/网侧实际），检查数据量是否合理：

- **验证窗口**：匹配日期往前推9天 ~ 匹配日期（共10天）
- 每天数据量范围应在 4~20 条之间

## 降级查询与日期连续性验证

**核心思想**：先找到"预期日期与实际最大日期一致"的offset，然后验证该日期往前推10天的数据是否完整。

**降级触发条件**：以下两种情况都会降级到下一个offset：
- 完全无数据（rows为空）
- 有数据但最大日期与预期不匹配

**降级查询流程**（以 `["D-4", "D-5", "D-6"]` 为例）：

```
第一优先：查 D-4
├─ 有数据，最大日期 == 预期 → ✅ 验证 D-14 ~ D-4 连续性
├─ 有数据，最大日期 != 预期 → 降级查 D-5
│   ├─ 有数据，最大日期 == 预期 → ✅ 验证 D-15 ~ D-5 连续性
│   ├─ 有数据，最大日期 != 预期 → 降级查 D-6
│   │   ├─ 有数据，最大日期 == 预期 → ✅ 验证 D-16 ~ D-6 连续性
│   │   ├─ 有数据，最大日期 != 预期 → ❌ 报告 "预期 X，实际 Y"
│   │   └─ 无数据 → ❌ 报告 "近10天无数据"
│   └─ 无数据 → 降级查 D-6（同上）
└─ 无数据 → 降级查 D-5（同上）
```

**连续性验证窗口**：

- 匹配成功（offset = D-N，最大日期 = D-N）→ 验证 **(D-N-9) ~ (D-N)** 共10天
- 例如：最终停在 D-6（5/27），则验证 **5/18 ~ 5/27** 的连续性
- 数据量校验使用相同的10天窗口

**全部无数据**：
- 所有 offset 的查询结果都为空 → 不跑连续性检查
- 直接报告 "近10天无数据 (X月X日 ~ X月X日)"

**关键规则**：
- 连续性验证范围始终为10天窗口（匹配日期往前推9天 ~ 匹配日期）
- 数据量合理性校验也使用相同的10天窗口
- 所有 offset 都无数据时，直接报告范围，不执行额外查询

## 注意事项

- 暂未接入的数据类型会被跳过，不进行校验
- 多时间值的数据类型，优先查询较新时间，无数据或日期不符时降级到较旧时间
- 断言失败时需明确指出预期与实际的差异
- 日期缺失时需列出具体的缺失日期
- `am.json` / `pm.json` 对应不同的调度任务（08:35 / 17:15），内容不同；改之前先确认改的是哪个文件
