## P1-2 多 temperature 模型

### 是什么

把 `models.py` 中的 `llm_model`（temperature=0.5）拆为 3 个：

```python
llm_model_generate = ChatOpenAI(..., temperature=0.3)  # 用例生成 - 适度创造性
llm_model_review = ChatOpenAI(..., temperature=0.1)    # 用例评审 - 高度确定性
llm_model_coverage = ChatOpenAI(..., temperature=0.1)  # 覆盖率 - 高度确定性
```

## P1-4 提示词补齐评审缺失的 3 个生成约束

### 是什么

`case_review_prompt.py` 7 个评审维度中，**维度 3（结果可验证）、维度 4（前置条件完整）、维度 5（测试数据有效）** 在 `generate_case_prompt.py` 的"约束规则"里没有对应规则。

需要在生成阶段补 3 条约束，避免"先生成宽松版 → 被评审打回"。

### 推荐取舍

| 方案          | 工时   | 收益               |
| ------------- | ------ | ------------------ |
| **A. 立即做** | 10 min | 中等，依赖后续验证 |

**我的建议**：先做（修起来简单），做完后跑端到端对比一轮通过率。

## P1-5 评审不通过用例的处置策略

### 是什么

当前 `case_review_prompt.py:19` 写"不通过的用例将被丢弃"，但 `case_generator.py:160-163` 实际存进 `review_failed_cases`，再无下游使用——**处置方式模糊**。

### 三个选项

| 选项                           | 说明                                  | 工时                 |
| ------------------------------ | ------------------------------------- | -------------------- |
| **A. 真的丢弃**                | 评审节点不返回 `_case`（返回空 dict） | 5 min                |
| **B. 落盘 `case_failed.json`** | `save_case` 节点同时落盘失败记录      | 10 min               |
| **C. 入库但不展示**            | 落库时 `status='failed'`，用户可查    | 30 min（涉及数据库） |

### 为什么待定

- 选项 A 最简单但丢失复盘数据
- 选项 B 平衡了简单和数据保留
- 选项 C 最完整但需要改数据库 schema

### 推荐取舍

**我的建议**：选项 B（落盘 case_failed.json），最低成本保留数据。

## P2-1 补充生成模式注入覆盖率上下文

### 是什么

`supplement_generate_case` 调用 LLM 时，**只传了 requirements / case_list / test_point**，没传当前覆盖率数值。LLM 不知道缺口有多大（30% vs 90% 影响优先级判断）。

### 为什么待定

- **支持做的理由**：让 LLM 感知覆盖率缺口大小，调整生成策略
- **不做的理由**：当前功能能跑，加这个是"锦上添花"

### 推荐取舍

**我的建议**：做（10 min 改 prompt + 函数签名），收益确定。

## P2-3 `case_id` 续编注入最大编号

### 是什么

`get_supplement_generate_case_prompt` 没传当前最大 `case_id`，LLM 自行起号易冲突。

在 HumanMessage 中追加 `**case_id 续编提示**：请从 TC-FUNC-{max_id+1} 起递增编号`。

### 为什么待定

- **支持做的理由**：避免 ID 冲突
- **不做的理由**：如果 case_id 是用 UUID 而不是 TC-FUNC-001 格式，冲突概率低

### 推荐取舍

**我的建议**：做（10 min），有明确收益。

## P2-4 评审并发限流

### 是什么

`_case_review_router` 用 Send 派发 N 个评审任务，SiliconFlow QPS 限制下 N 条用例并发容易触发 429。

### 三个选项

| 选项                                   | 说明                                       | 工时   |
| -------------------------------------- | ------------------------------------------ | ------ |
| **A. `asyncio.Semaphore(N)` 异步限流** | 把 `_review_case` 改为 async，加 semaphore | 30 min |
| **B. 同步分批**                        | router 把用例分批（每批 5 条），每批顺序跑 | 15 min |

### 为什么待定

- **A** 收益最大，但涉及 langgraph 节点 async 化
- **B** 简单可靠，但牺牲了部分并发

### 推荐取舍

**我的建议**：B（同步分批），风险最低。如未来 SiliconFlow 配额吃紧再升级到 A。

## P3-1 给子目录加 `__init__.py`

### 是什么

`pormpts/`、`workflow/`、`tools/`、`common/` 四个子目录均无 `__init__.py`。

```bash
touch lee/05用例生成agent/03用例生成&数据存储集成Agent/rag_agent/pormpts/__init__.py
touch lee/05用例生成agent/03用例生成&数据存储集成Agent/rag_agent/workflow/__init__.py
touch lee/05用例生成agent/03用例生成&数据存储集成Agent/rag_agent/tools/__init__.py
touch lee/05用例生成agent/03用例生成&数据存储集成Agent/rag_agent/common/__init__.py
```

### 为什么待定

- **支持做的理由**：Pycharm 静态检查会标红；跨平台移植时可能出问题
- **不做的理由**：当前 IDE 自动注入 path 跑得通

### 推荐取舍

**我的建议**：做（2 min），几乎零成本。

**额外建议**：把 `pormpts`（缺 m）改名为 `prompts` 是更彻底的修复，但属于破坏性改动（影响所有 import 路径），建议**先记录在 TODO**，等有维护窗口时再做。
